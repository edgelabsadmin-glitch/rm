# Design 08 — CEO View

**Phase:** 2 (Design); refreshed Phase 2.5 against Tier-0
**Tier:** 3 — late Phase 2 / extending into Phase 3

## Purpose

The CEO View is **the narrative surface for executive consumption**. PM_CONTEXT memory `project_pizzazz_feature` is unambiguous: **this is not a dashboard of RM activity** but an AI-RM speaking to the CEO in first person, weekly cadence. It is the visual-first surface (§6 design rule 16) that earns the CEO's attention.

**The CEO View is Pulse's most-brand-moment surface.** It is the place where Pulse "tells the story" to leadership weekly. Visual treatment is therefore the most purple-rich of any Pulse surface — every Tier-0 brand-moment token (`--color-brand-primary`, `--color-brand-primary-glow` tinted shadow, the conic-gradient health ring) gets used here, deliberately and without restraint.

**Visual register references:**
- `01_design/00_design_language.md` — Tier-0 tokens, the single source of truth.
- `01_design/00_design_language_preview.tsx` — the React preview's hero card is the closest analogue (the AI-RM voice paragraph above the conic ring).
- `rm-intelligence-agent/data/demo.html` — the inline-tag voice (`<num>`, `<bad>`, `<good>`, `<quote>`, `<em>`) originated here. The CEO View's prose pipeline is the same renderer.

---

## Inputs

- **Aggregated event log throughput** (Design 04): approved actions per skill / per RM / per week; outcome rates.
- **Account health rollups** (Design 07): tier distribution across the book; tier-change events in the week.
- **Per-account narratives** generated by an aggregator skill (CEO View has its own composer; the per-account narrative shape is lifted from `rm-intelligence-agent/src/generate_narratives.py`).
- **Cross-account pattern cards** from Skill 10.
- **Outcome stories** — concrete wins (action approved → outcome captured) selected as the week's highlights.

## Outputs

- **A weekly CEO View page** rendered at `/ceo` in the Pulse UI; readable in ≤5 minutes.
- **A weekly email digest** to the CEO (and CC list) — same content, single self-contained HTML email (echoing the demo.html fallback discipline).
- **An optional Markdown / PDF export** for offline reading or sharing in board materials.

**Design-language application — all references resolve through Tier-0:**

- **Page background:** `--color-surface-page` (#FAFAFA). Outer shell `--color-surface-chrome` (#F5F5F7) with `rounded-[2rem]` + `shadow-2xl shadow-slate-200/70` per Tier-0 §6 + §9.1.
- **The AI-RM voice block at the top is the visual hero of this surface.** It uses the hero-card treatment from Tier-0 §8.7: solid `--color-brand-primary` background, `--color-text-on-brand` text, `rounded-[2rem]` radius, `shadow-xl shadow-[--color-brand-primary-glow]` tinted shadow. The "Pulse on behalf of the book" signature is in `--color-text-on-brand-faint` (white/65), tiny, calm.
- **Voice prose:** rendered via the Tier-0 §10 inline-tag whitelist — `<num>` in `--font-mono` (JetBrains Mono), `<quote>` in Inter italic (Tier-0 confirmed humanist-sans family), `<bad>` in `--color-risk-high-fg`, `<good>` in `--color-risk-low-fg`, `<em>` in Inter italic slate-900. Same renderer as rm-intelligence-agent — the static-HTML fallback (§F below) consumes the same JSON.
- **Numbers** ("47 actions approved this week") use `<num>` styling — `--font-mono`, slightly darker than body.
- **Bad-signal callouts** in `<bad>` styling; good-signal in `<good>`.
- **Account health pulse chart** uses the Tier-0 signal-vector-bar primitive (§8.9) — `--color-surface-track` rail + `--color-brand-primary` fill, `h-2` rounded-full. One bar per tier (Healthy / Stable / Watch / At-Risk / Escalated). No D3, no zoomable views. Per §6 design rule 20 ("Outcome-led, not workflow-led").
- **Top 3 stories of the week** cards use the standard Tier-0 Card primitive (§8.3): `--color-surface-card` + `--color-border-subtle` + `rounded-3xl` + `shadow-sm`, padding `p-5`. Card eyebrows ("Pinnacle expansion · won 2026-05-18") use the metadata typography from Tier-0 §3 — `text-xs text-secondary`.
- **The book in numbers** strip uses the Tier-0 §8.7 hero-style sub-cards from the preview (the `pulse-facts` strip pattern) — `rounded-2xl border border-white/15 bg-white/10 px-3 py-3 text-xs text-white/85`. Each fact in `--color-text-on-brand-strip`. This strip lives *inside* the AI-RM voice hero, not as a separate card.
- **"What I'd ask of you" section** uses the Tier-0 trust-layer callout primitive (§8.13): white card + `--color-brand-primary-soft` border + `ShieldCheck` icon in `--color-brand-primary` + slate-500 body copy. Only renders when the composer decides there is a CEO-warranted ask (Q94).
- No "click to drill down" except where it doesn't break the narrative — movers (clickable Customer names) drill to the per-Account profile; numbers are clickable to a per-RM breakdown.
- **Single-page.** Scrollable on mobile; one fold's worth on desktop above the "What I'd ask of you" section.
- **Motion** per Tier-0 §7: page renders static on first load (no auto-playing motion); fade-and-lift only on user interaction (clicking a mover or expanding the brief).
- **No dark mode default.** The CEO View ships purple-rich on warm white. v1.5+ may add a dark-variant for export-to-PDF-on-dark-document scenarios.

### Agent-presence indicator — Pulse Bar (Breathing) integration

**Locked Session 10.** The indicator is the **Pulse Bar (Breathing)** — Tier-0 §8.14 + §6 design rule 22. Canonical render: `01_design/agent_presence_variants/04_pulse_bar_breathing.html`.

The CEO View shares the chrome with every other Pulse screen and inherits the Pulse Bar directly — the bar is visible at the top of the CEO View just as on every other screen. This is the variant's primary advantage over the deferred V3 (Account-Card Ambient Ring): the CEO View has no left account rail, so a rail-local indicator would go silent on this very surface. With the Pulse Bar, Pulse stays "alive" on the leadership-facing read.

Indicator behavior on the CEO View specifically:
- When Pulse is **composing the weekly narrative** (the Friday 16:00 generation job), the bar breathes — leadership can drop into the CEO View mid-composition and see Pulse actively working on the read.
- When the weekly narrative is **ready for review**, the bar fires its 600ms heartbeat and a "new weekly read" badge appears on the relevant header affordance.
- During normal reading of an already-composed weekly view, the bar holds at idle — calm baseline, Pulse present but not currently reasoning.

The bar is the indicator that signals "Pulse is alive" even when the operator surface (Action Queue) isn't in view.

### Cadence

- **Weekly,** generated every Friday 16:00 local for the week ending Friday.
- An additional **monthly** rollup runs on the 1st of each month, sent to the CEO and the VP of Client Success as a "month at EDGE" expanded view.
- Out of Phase 1 scope: daily / on-demand CEO views; in v1.5+ candidate list.

### Tone and voice spec

The CEO View speaks as **"Pulse, on behalf of the book of business"** — first-person plural where natural, candid, prioritizing color over completeness, never sycophantic, never "everything is fine" when it isn't. Specifically:

- **One thing the CEO might miss** is foregrounded each week (the lead AI-RM-voice paragraph).
- **Wins are credited to specific RMs by name.** ("Priya R. surfaced it the same day.")
- **Losses are named without finger-pointing.** ("Acrisure shifted Stable → Watch; CFO mandate to consolidate vendors.")
- **Concrete numbers always cited** in `<num>` styling so the CEO can challenge them.
- **No empty optimism.** If the week was quiet, the view says so in two sentences and ends.
- **The "What I'd ask of you" section appears only when warranted.** Skipped when there's no clear CEO ask.

### Personalization

The CEO View is the same content for the CEO and the VP of Client Success in Phase 1. A small per-recipient header line ("For Eddy" / "For [VP]") and (optionally) the "What I'd ask of you" section may differ. Multi-recipient personalization (different sections to different leaders) is v1.5+.

### Demo HTML fallback

Per Decision 12 + memory `Demo HTML fallback`, **the same composer can produce a single-file static HTML** rendered through the rm-intelligence-agent renderer (`render_demo.py`). This is the demo-fallback mode: if Pulse infrastructure is unavailable during a high-stakes presentation, the static HTML demo is preserved and works without any network.

The Phase 4 implementation makes this concrete: the composer emits a structured JSON, and **two renderers** consume it — (a) the live React/Pulse-UI renderer and (b) the static-HTML renderer (lifted from rm-intelligence-agent). Same content, two output paths.

## What this is NOT

- **Not a dashboard.** No KPI grid. No filters. No interactive controls beyond clickable account names + numbers.
- **Not a daily report.** Weekly is the cadence; monthly is the rollup. Daily / real-time is v1.5+.
- **Not RM-facing.** RMs have the Action Queue (Design 03) and the Per-Profile Markdown Layer (Design 06). The CEO View aggregates *above* the RM surface.
- **Not editable.** The CEO does not modify the view (it's a read surface). The "What I'd ask of you" section may, in v1.5+, become an inline-replyable surface.
- **Not Slack-delivered.** `feedback_dont_flood_slack`. Email + UI page are the delivery surfaces.
- **Not generated by the policy module.** Composition is a skill; policy governs *action proposals*. The CEO View is a read-only synthesis, not an action.
- **Not the place tier-aware behavior surfaces.** The CEO sees tier distribution but does not approve per-tier policy; that's in the Admin module (Design 09).
