# Skill 11 — detect-expansion-intent-from-job-posting

**Lifecycle stage:** expansion
**Phase:** 1 (added Session 10; locked Session 11)
**Tier-aware:** yes — SMB / Mid-Market / Enterprise variants per §6 rule 4
**Status:** spec backfilled Phase 3 (the skill's behavioral shape was scoped in Spike 4 §Q4; this spec captures the design-level form for completeness)

## Trigger

**Episode-driven.** Fires on every `episode-ingested` event where:
- `source == "opportunity-tracker"`, AND
- `content.match.tier ∈ {"hottest", "warm"}` (configurable; `general` suppressed by default per policy), AND
- `content.match.matched_role IS NOT NULL`, AND
- `content.match.work_arrangement != "on-site"` (post-precision-fix per Spike 4 §Q2).

`off-scope` tier postings never enter the memory layer (per Q120 disposition); they are persisted in `expansion_intent_signals` with `processed_status='skipped:off-scope'` and never produce Episodes, so Skill 11 cannot fire on them by construction.

## Inputs

- **Retrievers required:** `get_customer_context(customer_id)` (Design 01) — for placed-talent count, recent customer health, recent signals.
- **Per-Profile Markdown read:** Customer profile (Design 06) — for stakeholder context to personalize the proposed outreach.
- **External calls (READ ONLY):** none beyond the retriever.
- **Episode content:** `match.{tier, matched_role, score, reasoning, outreach_suggestion, signals, work_arrangement}` + posting metadata.
- **Policy inputs:** Customer's `Segment__c` tier.

## Behavior

For each ingested opportunity-tracker Episode that passes the trigger:

1. **Read the Customer's context bundle** (placed-talent count, current health tier from Design 07, recent signals last 30 days).
2. **Cross-reference with `expansion_signal_verbal_capacity_mention_v1`.** If the same customer has verbal expansion mentions in the last 30 days for the same role family, **compose into a single Action Queue card** rather than emitting two (per Q134 disposition). The verbal mention strengthens the posting-match signal; one card with combined evidence.
3. **Drafted outreach email** — lift `match.outreach_suggestion` (already LLM-generated by opportunity-tracker), polish against the Customer profile's Communication preferences and Stakeholders sections.
4. **Drafted SFDC Task** for the account RM with full evidence citation (posting URLs, matched role, current placement count, outreach summary).
5. **Tier-aware envelope** per §"Tier variants" below.
6. **Emit `action-suggested` event** with the inline-tag-voice reasoning capture (Design 04).

The skill **does not write to Salesforce directly** (per §6 rule 6); writes happen via the Action Queue dispatch handlers on approval.

**Reasoning structure** (Design 04 §"Reasoning capture"):

```
[skill: detect-expansion-intent-from-job-posting]
[context: Customer=<name>, tier=<segment>, posting_count=<N>, matched_role=<role>]

Signals consulted:
  - <num>N</num> opportunity-tracker matches for <matched_role> in last <window> days
  - opportunity-tracker tier: <tier>; aggregate score: <score>
  - <num>M</num> Active placements at this customer
  - Customer health: <tier>
  - Verbal capacity mention from <date> (if composed)
  - Cross-account pattern (if Skill 10 found similar matches at related customers)

Reasoning:
  <Customer> is hiring <role> roles externally despite Edge having <M> active
  placements there. <good>Strong expansion signal</good> — the outreach
  proposes a conversation about expanding the existing placement footprint
  before they fill externally.

Proposed action: email draft to <champion> + SFDC Task for RM.
```

## Guardrails

- **No customer outreach within 14 days of last Skill 11 action on this customer.** Prevents same-customer flood when opportunity-tracker surfaces multiple matches in close succession. Rate-limit table keyed on (customer_id, week).
- **Cross-coordination with Skill 06 (advocacy).** If the customer is already in an active advocacy motion (Skill 06 action proposed in last 7 days), Skill 11 defers — advocacy customers get a single coordinated approach, not two.
- **Composition with `expansion_signal_verbal_capacity_mention_v1`.** When both fire, single card (per Q134). The why_oneline cites both pieces of evidence.
- **Off-scope postings invisible.** By construction — they don't enter the memory layer.
- **No reference to opportunity-tracker by name in user-facing copy** (white-label rule, §6 rule 1). The action card may say "we noticed multiple recent job postings for [role]" — never "opportunity-tracker matched."

## Output / Proposed action shape

```yaml
action_type: expansion-intent-outreach
delivery_channel: email + sfdc_task
body:
  email_draft:
    to: <champion_contact_email>
    cc: [<additional_contacts>]                # Mid+; Enterprise also cc's VP-CS
    subject: <generated from matched_role + customer_name>
    body: <inline-tag-rendered draft built from match.outreach_suggestion + Customer profile color>
  sfdc_task:
    subject: "Expansion opportunity: <matched_role> at <customer_name>"
    description: <full citation — posting URLs, match.reasoning, current placement context>
    related_to: <Account.Id>

modifiable_fields: [body.email_draft.body, body.email_draft.subject,
                     body.email_draft.cc, body.sfdc_task.description, body.sfdc_task.due_date]
```

## Tier variants

| Customer tier | Variant |
|---|---|
| **SMB** | `warm`-tier match → auto-approve at +2h delay (low blast radius). `hottest`-tier → human-required. |
| **Mid-Market** | All tiers human-required. RM gets the drafted email + SFDC Task. |
| **Enterprise** | All tiers human-required + cc VP-CS on the email draft. **Static EBR-tie-in copy attached** per Mitigation B (Spike 4 + Decision 41) — "We noticed your hiring activity in [role family]; happy to walk through your staffing forecast at our next EBR." Dynamic-date insertion is v1.5+ (§12 #14). |

Tier classification reads from `Account.Segment__c` (Q22 / Q123 resolved).

## Outcome detection

- **Opportunity created** in SFDC for this customer within 30 days of action dispatch → `outcome-recorded` type `expansion-opportunity-created`. The strongest success signal.
- **Email reply** received within 14 days → `outcome-recorded` type `customer-responsive`.
- **No movement** after 30 days → `outcome-missing`. Re-queue is *not* automatic (Phase 1 — rate-limit prevents same-customer re-fire for 14 days; if no new postings surface, no new action).

## What this is NOT

- **Not the matcher.** opportunity-tracker does the matching (rules + LLM); Skill 11 consumes the result.
- **Not the integration adapter.** The opportunity-tracker Signal Source Adapter (spec 015) handles the Postgres polling and Episode normalization; Skill 11 fires on the resulting `episode-ingested` events.
- **Not a write to SFDC.** Per §6 rule 6, writes happen only via Action Queue dispatch on approval.
- **Not user-facing-aware of opportunity-tracker.** White-label rule.
